## Overview

Sometimes you may need to access data inside your model which you do not wish to hard-code inside the model code. For example, if you are fetching data from an external service, you may have an API key which you need to use as part of the query to identify you as an authorised user of the service. You should not store the API key inside the model code, as in the future the API key may be updated, and if this were to happen, the old versions of your model with the old API key would no longer work.

hx Renew allows you to store these pieces of data as 'secrets'. These are data items which are stored separately from the model files, and which may be updated independently from the model. These secrets can be assigned to certain models, and will be available for access by a model's Async Task code. Whenever the Async Task requests a secret, it will always receive the latest version of the secret, so when you update the secret all previous versions of the model will be updated to use the latest value.

Optionally, for a single secret, you may have three different values that will be passed into policy options depending on the context.
If you do not require such granularity and would prefer to use a single value across all use cases, simply provide a Production Value.

## Creating a Secret

The secrets page can be found by navigating to the Developer Dashboard and clicking 'Secrets' on the left menu.

To create a Secret:

1. Click on the "Create Secret" button in the top right corner of the screen

<img
  src={require("../../assets/ModelDevelopmentProcess/Models/secrets_dashboard.png").default}
  width="1000"
/>

2. Fill in the details of the new secret

<img
  src={require("../../assets/ModelDevelopmentProcess/Models/secrets_create_secret.png").default}
  width="1000"
/>

- **Identifier**
: Used to refer to the secret in the model code. It must be a valid Python
variable name, meaning it must begin with a letter and only contain
alphanumeric characters and underscores. Additionally it must not be any of
the reserved Python keywords (e.g. `if`, `elif`, `false`, etc.) <a
href="https://docs.python.org/3.9/reference/lexical_analysis.html#keywords"
target="_blank">**Full list of reserved Python keywords**</a>.

- **Friendly Name**
: A short descriptive name for the secret.

- **Description**
: A longer description of the secret.

- **Production Value** is mandatory and will be used for live options, as well as serving as a fallback in scenarios where the Test or Development Value is not provided. This can be any string.

- **Test Value**, if provided, will be utilized by policy options that are enabled for [testing](/platform/developer/model-development-process/models/publishing-a-model.mdx#releasing-a-version-of-a-model-for-testing).

- **Development Value**, if provided, will be used by the model viewer in the development portal.

- **Visibility**
: Secrets can be made available to tasks in models, to custom UI component code in the view, or to both. The default is that they are available to tasks in models only.

## Creating a New Version

When retrieving data from secrets, a model will always get the latest version. Therefore to update the value of the secret, we need to create a new version of the secret. To do this:

1. Navigate to the secret details page.

2. Here you can enter new secret values, click 'Update keys' to apply.

<img
  src={require("../../assets/ModelDevelopmentProcess/Models/secret_new_version.png").default}
  width="1000"
/>

## Changing a Secret's Permissions

To modify which models have access to your secret:

1. Navigate to the secrets.

2. Scroll down to the 'Model Permissions' section. This shows a list of models that have access to the secret.

<img
  src={require("../../assets/ModelDevelopmentProcess/Models/secret_model_permissions.png").default}
